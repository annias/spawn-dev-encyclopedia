    GAME DEVELOPMENT LANGUAGE & CONCEPTS ENCYCLOPEDIA

    Chapter 9 — QA, Testing & Release Terminology

    The Core Mental Shift: What QA Actually Means

    QA (Quality Assurance) is often misunderstood as:

    “Finding bugs.”

    In professional game development, QA is:

    The systematic process of validating that a game is stable, consistent, reproducible, and compliant across all expected player behaviors and hardware environments.

    A key AAA insight:

    “A game is not finished when it works — it is finished when it cannot reasonably break.”

    QA is not a final phase. It is a continuous feedback system embedded into development.

    Chapter 1: QA Philosophy in Game Development

    Quality Assurance (QA)

    The discipline responsible for:

    • Detecting defects
    • Verifying fixes
    • Ensuring stability
    • Validating player experience consistency

    Quality Control (QC)

    Subset of QA focused on:

    • Testing outputs
    • Verifying correctness of builds

    Test Case

    A structured scenario used to validate functionality.

    Example:

    • “Player dies in lava → respawn at checkpoint”

    Test Suite

    Collection of test cases covering a system.

    Regression Testing

    Testing that previously fixed issues have not returned.

    Why regression testing matters

    Games are highly interconnected systems.

    Fixing one issue often breaks another.

    Chapter 2: Bug Classification Language

    Bug

    Any unintended behavior.

    Defect

    Formal term used in production tracking systems.

    Issue

    General umbrella term (bug, task, improvement).

    Repro Steps (Reproduction Steps)

    Exact steps to reliably trigger a bug.

    Example:

    1. Start mission2. Equip weapon3. Enter combat4. Reload → crash occurs

    Severity vs Priority (Critical distinction)

    Severity

    How bad the bug is technically.

    Priority

    How urgently it must be fixed.

    Severity Levels

    S1 — Critical

    • Game crashes
    • Save corruption
    • Progress blocking

    S2 — Major

    • Gameplay broken but not blocking

    S3 — Minor

    • Visual glitches
    • UI issues

    S4 — Cosmetic

    • Slight animation or text issues

    Priority Levels

    P0

    Fix immediately.

    P1

    Fix soon.

    P2

    Fix when possible.

    P3

    Backlog.

    Edge Case

    Rare or extreme scenario not typical in normal gameplay.

    Example:

    • Player exploits physics to leave map bounds

    Corner Case

    Very specific condition where multiple systems interact unexpectedly.

    Chapter 3: Testing Methodologies

    Playtesting

    Observing real players interacting with the game.

    Focus:

    • Confusion points
    • Difficulty spikes
    • Unintended behavior

    Internal Testing

    Developers testing their own systems.

    External Testing

    Players outside the development team (often early access or focus groups).

    Smoke Testing

    Quick validation that the build launches and core systems function.

    Stress Testing

    Pushing systems beyond expected limits.

    Example:

    • Hundreds of enemies on screen
    • Extreme network load

    Load Testing

    Measuring performance under heavy usage conditions.

    Compatibility Testing

    Ensuring the game runs on different:

    • Hardware
    • Consoles
    • Drivers
    • Operating systems

    Chapter 4: Build and Release Language

    Build

    A compiled playable version of the game.

    Internal Build

    Used by developers only.

    External Build

    Used by testers or players.

    Nightly Build

    Automated daily build from latest code.

    Candidate Build (RC)

    Potential release version.

    Gold Master (GM)

    Final approved release build.

    Patch

    Post-release update.

    Hotfix

    Urgent patch for critical issues.

    Update vs Patch

    • Patch = fixes
    • Update = fixes + content changes

    Chapter 5: Certification & Platform Compliance

    Certification

    Approval process required by platform holders before release.

    Example platforms:

    • PlayStation
    • Xbox
    • Nintendo
    • Steam (less strict but still regulated)

    TRC (Technical Requirement Checklist)

    Platform-specific rules a game must follow.

    Example:

    • No crashes during suspend/resume
    • Proper save handling

    Compliance Testing

    Ensuring game meets platform rules.

    Submission

    Process of sending a build to platform for approval.

    Failure in Certification

    Common causes:

    • Crash bugs
    • Save corruption
    • UI blocking input
    • System violations

    Chapter 6: Debugging & Diagnostic Language

    Crash

    Game or application terminates unexpectedly.

    Freeze

    Game stops responding but does not crash.

    Hang

    Similar to freeze; system stuck in infinite loop or blocked state.

    Null Reference Exception

    Trying to access non-existent object.

    Memory Leak

    Memory not released after use, causing performance degradation.

    Race Condition

    Timing-dependent bug in concurrent systems.

    Example:

    Two systems updating same variable simultaneously.

    Desync

    Multiplayer state mismatch between clients and server.

    Chapter 7: Performance QA Language

    Frame Drop

    Sudden decrease in FPS.

    Stutter

    Irregular frame pacing causing jitter.

    Hitching

    Brief freeze due to asset loading or CPU spikes.

    Frame Time Spike

    Sudden increase in time needed to render a frame.

    Bottleneck

    System limiting overall performance.

    Overhead

    Extra computational cost not directly tied to gameplay.

    Chapter 8: QA Workflow Systems

    Bug Database

    Central system tracking all issues.

    Common tools:

    • Jira
    • Hansoft
    • Trello (smaller teams)

    Ticket

    Individual bug or task entry.

    Status Flow

    Typical states:

    New → Assigned → In Progress → Fixed → Verified → Closed

    Verification

    QA confirms fix is correct.

    Reopen

    Bug returns after being marked fixed.

    Chapter 9: QA Metrics

    Bug Density

    Number of bugs per system or area.

    Escape Rate

    Bugs that reach players after release.

    Test Coverage

    Percentage of system tested.

    Stability Metrics

    Crash rate, freeze rate, etc.

    Chapter 10: Live Build Testing

    Build Verification Test (BVT)

    Ensures build is stable enough for testing.

    Continuous Integration Testing

    Automated testing during development.

    Automated Testing

    Scripts simulate gameplay behavior.

    Used for:

    • Regression checks
    • Repeated validation

    Chapter 11: Player-Facing QA Concepts

    UX Bug

    Confusing or misleading experience, not a technical failure.

    Example:

    • Players don’t understand UI icon meaning

    Gameplay Bug

    Mechanics not functioning as intended.

    Visual Bug

    Graphical issues.

    Audio Bug

    Missing or incorrect sound playback.

    Soft Lock

    Player cannot progress but game still runs.

    Example:

    Quest breaks leaving player stuck.

    Hard Lock

    Game becomes completely unresponsive.

    Chapter 12: Senior QA Mental Model

    Beginner Thinking

    “Does the game have bugs?”

    Intermediate Thinking

    “Where are the bugs and how do we fix them?”

    Senior Thinking

    “What systemic patterns in design, architecture, or production are generating instability?”

    Expert Thinking

    “How do we design systems that are inherently testable, observable, and resilient under unpredictable player behavior?”

    Key Insight

    In professional studios:

    QA is not the end of development — it is the feedback system that keeps the entire development process grounded in reality.

    Without QA:

    • Design assumptions fail
    • Systems drift into instability
    • Player experience becomes unpredictable
